# Integration Test Skill — Bootstrap Template

## Bootstrap: questions for the operator

When the operator wants to install `/integration-test`, ask these questions **before**
generating SKILL.md:

```
For the /integration-test skill I need a quick note on:

1. What are your 3–5 critical tier-1 checks?
   (Tier-1 = blocks issue-close if red)
   Examples: "API reachable", "Database writes", "Main daemon running",
             "Latest signal < 10 min old", "No open error in log"

2. What are optional tier-2 checks?
   (Tier-2 = warning, not a blocker)
   Examples: "Log size < 100 MB", "Config version in sync", "Backup current"

3. Should /integration-test run automatically after every /implement?
   (Yes → add hint to implement/SKILL.md / No → manual invocation)
```

Wait for answers. Then generate SKILL.md from the skeleton below and write it to
`{PROJECT_PATH}/.claude/skills/integration-test/SKILL.md`.
Replace placeholders:
- `{{PROJECT_NAME}}` → from phase 0 info-gathering
- `{{TIER1_CHECKS}}` → answer 1 as bullet list
- `{{TIER2_CHECKS}}` → answer 2 as bullet list
- `{{POST_IMPLEMENT}}` → "Yes" or "No"

**If post-implement = yes:** add a hint at the end of `.claude/skills/implement/SKILL.md`:
"After completion: run /integration-test and check the result."

# /integration-test — System integration tests

**Triggers:** after every /implement run, after breakfix fixes, or when the system
behaves strangely.

**Goal:** quickly check whether the system as a whole works correctly — not whether
individual code units are right (that's unit testing).

## Tier concept

| Tier | Meaning | On failure |
|------|---------|-----------|
| **Tier-1** | Critical — core function broken | Do NOT close issue, run /breakfix immediately |
| **Tier-2** | Warning — secondary function affected | Check + document, no hard stop |

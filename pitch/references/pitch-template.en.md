# Copy template: `pitch/PITCH-XX.md`

This file is generated by the `/pitch` skill at the end of the 6-step session and stored under `pitch/PITCH-XX.md` in the project repo (XX = next free number with leading zeros, e.g. `PITCH-12.md`). The file is committed, NOT gitignored — pitch briefings are part of the project history.

---

## Frontmatter schema

```yaml
---
pitch_id: PITCH-12
sprint: 12
created_at: 2026-04-28T14:00:00Z
related_intents: [INTENT-3, INTENT-5]
related_stories: [BOO-15, BOO-16, BOO-17]
metrics_snapshot:
  loc_delta: "+2,341 / -890"
  coverage_trend: "82% → 84% (+2pp)"
  p95_change: "180ms → 145ms (-19%)"
  iterations_avg: 2.3
  feature_flags_active: 3
  intent_fulfillment_score: 0.85
demo_path: "User-Onboarding → Search → Checkout"
status: prepared | delivered | post-mortem
---
```

## Field reference

| Field | Description |
|---|---|
| `pitch_id` | Running ID, format `PITCH-XX` with leading zeros. Highest number in `pitch/PITCH-*.md` + 1. |
| `sprint` | Sprint number or sprint-file reference the pitch covers. |
| `created_at` | ISO-8601 UTC timestamp of creation. Used in step 3 to back-translate for the architecture diff. |
| `related_intents` | List of intent IDs from `intents/INTENT-XX.md` that are in pitch focus. Multiple allowed. |
| `related_stories` | List of Linear story IDs that will be presented. |
| `metrics_snapshot.loc_delta` | Sum of `+/-` lines from `git log --shortstat --since=<sprint-start>`. Format: `"+X / -Y"`. |
| `metrics_snapshot.coverage_trend` | Test coverage movement since last pitch. Source: sprint file or latest CI report. Format: `"before% → after% (+/-Xpp)"`. |
| `metrics_snapshot.p95_change` | p95 latency movement since last pitch. Source: performance baseline (BOO-16). Format: `"beforeMs → afterMs (+/-X%)"`. |
| `metrics_snapshot.iterations_avg` | Average iteration count per story in scope. Source: `meta.json` from local reports. |
| `metrics_snapshot.feature_flags_active` | Number of active feature flags. Source: `.claude/feature-flags.json` (BOO-17). |
| `metrics_snapshot.intent_fulfillment_score` | Aggregate score 0–1 across all `related_intents`. Computed in step 4. |
| `demo_path` | User journey in arrow notation (e.g. `"A → B → C"`). Suggestion from demo-path heuristic (step 4); operator may override. |
| `status` | `prepared` on creation, `delivered` after step 6, `post-mortem` if the pitch went sideways. |

## Body sections

Order is fixed, all sections are mandatory — mark empty sections with `_(none)_` instead of removing them.

### 1. Architecture diff since last pitch

From step 3 of the skill. Bullet list, per change:

- §-number from `ARCHITECTURE_DESIGN.md`
- What changed (1 sentence)
- Rationale from commits (1 sentence, optionally commit hash)

For the first pitch: full snapshot of `ARCHITECTURE_DESIGN.md` summarised, 3–5 bullets per §-section.

### 2. Quality-gate status

From sprint file and SonarQube (if enabled). Three sub-points:

- **Open findings** — count + severity distribution
- **Closed hotspots** — what was resolved during the sprint
- **Coverage movement** — before/after in percentage points

### 3. Intent fulfillment

One block per `related_intent`:

```markdown
#### INTENT-X — <short title>

- **Success criterion:** <verbatim from intents/INTENT-X.md>
- **Current state:** <measured value from sprint reports + feature flag state>
- **Score:** 0.X
```

Score is computed in step 4 of the skill. For criteria that cannot be measured: `Score: null — manual operator score required`.

### 4. Demo path suggestion

User journey from step 4 with 3–5 stations. Per station:

- **Station** — component/endpoint
- **Why** — 1 sentence on why it suits the demo (change intensity OR intent relevance)

The operator can freely edit the path during review (step 5) — see `demo-path-heuristic.en.md` §Operator override.

### 5. Open questions

What the skill doesn't know, what stakeholders might ask. Bullet list, 1 sentence per question. The skill may state assumptions but must mark them explicitly as such.

### 6. Outcome (post-pitch)

Empty on creation (step 5). Filled by the operator after the stakeholder meeting in step 6 — free text, 1–3 sentences: who attended, which questions came up, which decision was made. NO auto-generation.

# PITCH-12 — Sprint 12 stakeholder briefing

## 1. Architecture diff since last pitch

- §3.2 Search layer: switched from in-memory index to Elasticsearch cluster. Rationale: p95 below 150ms with >100k documents (commit a1b2c3d).
- §4.1 Checkout flow: added Stripe webhook retry logic. Rationale: idempotency for payment confirmations (commit e4f5g6h).

## 2. Quality-gate status

- **Open findings:** 4 (1 high, 3 medium)
- **Closed hotspots:** 2 SQL injection paths in the search endpoint
- **Coverage movement:** 82% → 84% (+2pp), focus in the checkout module

## 3. Intent fulfillment

#### INTENT-3 — Search latency below 150ms

- **Success criterion:** p95 for search queries drops from 180ms to <150ms within 2 months.
- **Current state:** p95 = 145ms (measurement 2026-04-27, CI report run-892)
- **Score:** 0.95

#### INTENT-5 — Halve the onboarding drop-off rate

- **Success criterion:** Drop-off rate in step 2 falls from 38% to <19% by sprint 14.
- **Current state:** Currently 27% (telemetry 2026-04-27), feature flag `onboarding-v2` rollout at 50%.
- **Score:** 0.62

## 4. Demo path suggestion

- **User-Onboarding** — account setup with new step-2 flow (largest LOC delta in sprint, covers INTENT-5).
- **Search** — search across a product list (shows the p95 improvement live, covers INTENT-3).
- **Checkout** — payment via the Stripe retry path (showcases §4.1 architecture diff).

## 5. Open questions

- When will the `onboarding-v2` feature flag be set to 100%? (Assumption: after sprint 13, depending on the drop-off trend.)
- How will the ES cluster scale once document count exceeds 500k? (Assumption: sharding, but not in sprint 12 scope.)
- Stakeholders may ask about the cost of the ES migration — numbers are not in the briefing.
